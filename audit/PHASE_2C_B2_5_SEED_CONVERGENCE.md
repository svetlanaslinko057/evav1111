## What the replay does

For each row in legacy `dev_wallets`, the script classifies the shape:

| Shape | Detection | Action |
|---|---|---|
| `orphan` | `available_balance + withdrawn_lifetime` set, no `paid_lifetime` | **skip** — preserve `mock_orphan` |
| `pool_seed` | `paid_lifetime` set, `withdrawn_lifetime` not set | normalise legacy + bridge replay |
| `already_converged` | normalised + `paid_lifetime` still present | **skip** — idempotent no-op |
| `unknown` | anything else | skip with warning |

For each `pool_seed` row:

1. **Normalise legacy shape** (preserves `earned_lifetime` exactly):
   ```
   available_balance  = earned_lifetime - paid_lifetime
   withdrawn_lifetime = paid_lifetime
   pending_withdrawal = 0      (folded into `available`)
   ```
2. **Bridge replay** through `MoneyService`:
   ```
   hold_escrow   (project_id=seed_bridge_<dev>, earned, fee_cents=0)
   release_escrow(project_id=seed_bridge_<dev>, earned, dev, fee=0)
   if paid > 0:
       process_payout(dev, paid, payout_batch_id=seed_bridge_<dev>)
   ```

Idempotency keys are deterministic (`seed_bridge_replay:<dev>:<leg>`),
so re-running the script appends nothing new to `money_ledger_events`.

## Honest disclosure

The script performs ONE direct legacy write per pool dev (the shape
normalisation). This is required because the seed leaves
`available_balance` / `withdrawn_lifetime` fields unset, which makes
`legacy_earned ≠ legacy_available + legacy_withdrawn` and is therefore
internally inconsistent — no projection can ever match it without first
splitting the breakdown.

The normalisation preserves `earned_lifetime` bit-for-bit; only the
breakdown fields the seed left unset are populated. It is reshaping, not
repair. The user's "no synthetic repair" rule is honoured: the ledger
entries themselves are exclusively produced by real `MoneyService`
operations (`hold_escrow`, `release_escrow`, `process_payout`), never by
hand-rolled adjustments.

## Why a separate script (not a seed patch)

The user explicitly listed both options:
> переписать seed_money_demo.py на bridge-based flows
> ИЛИ
> сделать отдельный replay_seed_money.py …

Option B (separate replay) was chosen because:

1. **No timing race with backend startup.** The seed runs before
   `_money_bridge_init()` registers its startup hook in some boot
   orderings; calling `MoneyService` from inside the seed risks a
   half-initialised repo and partial writes.
2. **Reversible.** If the replay is later determined to be wrong, the
   30 ledger rows are trivially removable by `actor=seed_bridge_replay`.
   A seed patch baked into `server.py` would be far harder to roll back.
3. **One-shot operation with explicit dry-run.** Operators see the full
   plan before any write happens (`--dry-run` is the only first-run
   default that matters for human review).
